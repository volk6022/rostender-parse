# Единый план развития и оптимизации Rostender Parser

Документ объединяет план расширения fallback-системы для внешних площадок (ETP GPB, Rosatom, Roseltorg, EIS) и план технической оптимизации производительности, памяти и качества кода.

---

## Этап 1: Архитектура и Общие Компоненты (Core & DRY)

Цель: Подготовить фундамент для новых функций и устранить дублирование кода.

### 1.1. Рефакторинг навигации и поиска

**Файлы:** `src/scraper/common.py` (новый), `src/scraper/active_tenders.py`, `src/scraper/historical_search.py`

**Действия:**
- Создать `src/scraper/common.py` и перенести в него общие функции:
  - `_fill_common_filters` (заполнение фильтров цены, ключевых слов, Chosen-select)
  - `_navigate_to_search` (переход на страницу поиска)
  - `submit_search` (отправка формы поиска)
- **Smart Navigation**: Добавить проверку `page.url` внутри `_navigate_to_search` перед выполнением `goto`, чтобы избежать навигации на ту же страницу.
- Обновить `active_tenders.py` и `historical_search.py` для использования общих хелперов из `common.py`.

### 1.2. Улучшение отказоустойчивости браузера

**Файл:** `src/scraper/browser.py`

**Действия:**
- **Exponential Backoff**: Переписать логику повтора в `safe_goto`. Заменить линейную задержку на экспоненциальную (2с, 4с, 8с, 16с) для обработки временных сетевых проблем и rate-limiting.
- **Standardized Recovery**: Обеспечить консистентный вызов `_recover_page` (сброс страницы в `about:blank`) во всех блоках `try...except` во всех stages.

### 1.3. Конфигурация учетных данных

**Файлы:** `src/config.py`, `config.yaml.example`

**Действия:**
- Расширить схему конфигурации: добавить секцию `credentials` с полями для каждой платформы:
  ```yaml
  credentials:
    eis:
      login: ""
      password: ""
    gpb:
      login: ""
      password: ""
    rosatom:
      login: ""
      password: ""
    roseltorg:
      login: ""
      password: ""
  ```
- В классе `Config` добавить метод `get_credentials(platform_name: str) -> dict | None`, который возвращает словарь `{login, password}` или `None`, если учетные данные не указаны.

## Этап 2: Оптимизация работы с данными (Database & Memory)

Цель: Повысить производительность при больших объемах данных и снизить потребление памяти.

### 2.1. Оптимизация операций с базой данных

**Файл:** `src/db/`

**Действия:**
- **Persistent Connections**:
  - Модифицировать `get_connection`: если передан аргумент с существующим соединением, использовать его. Если нет — создавать новое, но возвращать кортеж `(connection, needs_close)`.
  - В рамках `run_search_active` и `run_analyze_history` использовать одно соединение на весь запуск stage, передавая его между функциями.
- **Batch Upserts**:
  - Реализовать функцию `upsert_tenders_batch(conn, tenders: list[dict])` с использованием `executemany`.
  - Реализовать функцию `upsert_customers_batch(conn, customers: list[dict])` с использованием `executemany`.
- **Transaction Grouping**:
  - В циклах обработки результатов поиска заменить `commit()` после каждой записи на `commit()` каждые N итераций (например, 10–20 записей) или один раз после завершения обработки страницы результатов.

### 2.2. Атомарная экстракция данных через page.evaluate()

**Файлы:** Функции извлечения данных в `src/scraper/` (точные имена нужно уточнить после анализа кода)

**Действия:**
- Переписать функции, которые сейчас делают `page.content()` + Python regex, на `page.evaluate()` с JavaScript-скриптами внутри браузера.
- Заменить получение всего HTML-контента страницы на прямые запросы к DOM:
  - **`extract_inn_from_page`**: JS-скрипт ищет ИНН в тексте страницы или по селекторам и возвращает готовое значение.
  - **`get_customer_name`**: JS-скрипт находит название заказчика по текстовым маркерам.
  - **Сбор source_urls**: JS-проход по DOM собирает все ссылки, подходящие под паттерны внешних площадок, и возвращает готовый массив.
- Цель: минимизировать передачу больших объемов данных между браузером и Python-процессом, снизить использование RAM.

## Этап 3: Расширение системы Fallback (Новые площадки)

Цель: Реализовать извлечение ИНН и протоколов с внешних площадок (GPB, Rosatom, Roseltorg, EIS).

### 3.1. Регистрация доменов источников

**Файл:** `src/scraper/source_links.py`

**Действия:**
- Расширить словарь `SOURCE_DOMAINS`:
  - `new.etpgpb.ru` -> `gpb`
  - `zakupki.rosatom.ru` -> `rosatom`
  - `www.roseltorg.ru` -> `roseltorg`
  - `zakupki.gov.ru` -> `eis` (убедиться, что уже есть)

### 3.2. Слой аутентификации

**Файл:** `src/scraper/auth.py`

**Действия:**
- Реализовать функции входа на каждую платформу:
  - `login_to_gpb(page, credentials)`
  - `login_to_rosatom(page, credentials)`
  - `login_to_roseltorg(page, credentials)`
- Каждая функция должна:
  - Проверять наличие `credentials` перед попыткой входа.
  - Использовать робастные селекторы для полей ввода логина/пароля и кнопки входа.
  - Обрабатывать случаи, когда форма входа не найдена (тендер в публичном доступе).

### 3.3. Платформенные модули fallback

**Новые файлы:**
- `src/scraper/gpb_fallback.py`
- `src/scraper/rosatom_fallback.py`
- `src/scraper/roseltorg_fallback.py`

**Для каждого модуля реализовать:**
- `extract_inn_from_[source](page, url)`: Скрапинг ИНН заказчика со страницы тендера.
- `get_protocol_links_from_[source](page, url)`: Поиск ссылок на протоколы (публичные и私有ные/требующие входа).
- `download_protocol_from_[source](page, url, save_dir, ...)`: Скачивание файлов протоколов в директорию `downloads/{inn}/{tender_id}/{source}/`.

## Этап 4: Интеграция Fallback в пайплайн обработки

Цель: Связать новые модули с основным процессом обработки тендеров.

### 4.1. Интеграция в Стадию 1 (Поиск ИНН)

**Файл:** `src/stages/search_active.py`

**Действия:**
- Расширить цепочку fallback для поиска ИНН:
  - Получить все `source_urls` из тендера через `src.scraper.source_links.extract_source_urls`.
  - Пройтись по доступным источникам в порядке приоритета.
  - Для каждого источника вызвать соответствующую функцию `fallback_extract_inn`.
  - Остановиться при первом успешном извлечении ИНН.

### 4.2. Интеграция в Стадию 4 (Анализ протоколов)

**Файл:** `src/parser/html_protocol.py`

**Действия:**
- Переименовать `_try_eis_protocol` в `_try_external_fallbacks`.
- Обобщить логику для поддержки нескольких платформ:
  1. Для каждой внешней ссылки из данных тендера определить тип платформы.
  2. Если есть учетные данные для этой платформы — выполнить вход.
  3. Перейти на страницу тендера на внешней площадке.
  4. Найти и скачать доступные протоколы.
- Убедиться, что `ProtocolParseResult` корректно определяет `parse_source` (например, `rosatom_html`, `gpb_pdf`).
