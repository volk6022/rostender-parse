# План рефакторинга парсинга протоколов

## Цель

Улучшить качество определения количества участников (заявок) в протоколах тендеров, особенно в случаях:
- Наличия нескольких таблиц в одном протоколе
- Необходимости дедупликации заявок по номерам между разными таблицами
- Правильной идентификации таблиц с заявками по их названиям

## Этап 2: Дедупликация заявок по номерам

### 2.1. Извлечение номеров заявок из таблиц

**Проблема:** В разных таблицах одного протокола (или разных протоколов) могут быть заявки с одинаковыми номерами.

**Решение:**

```python
def extract_application_numbers_from_table(table) -> list[int]:
    """Извлекает номера заявок из таблицы.
    
    Ожидает первый столбец - номер заявки (1, 2, 3...)
    """
    numbers = []
    for row in table.rows[1:]:  # Пропустить заголовок
        first_cell = row.cells[0].text.strip()
        # Попытка извлечь номер
        match = re.match(r"^(\d+)", first_cell)
        if match:
            numbers.append(int(match.group(1)))
    return numbers
```

### 2.2. Агрегация заявок по тендеру

**Проблема:** Один тендер = несколько протоколов = разные таблицы.

**Решение:** Создать структуру для хранения всех заявок:

```python
@dataclass
class TenderApplications:
    tender_id: str
    applications: set[int]  # Уникальные номера заявок
    
    def add_from_table(self, numbers: list[int]):
        self.applications.update(numbers)
    
    def get_unique_count(self) -> int:
        return len(self.applications)
```

## Этап 4: Улучшение извлечения из текста (regex)

### 4.1. Приоритизация паттернов

Текущий порядок не оптимален. Предлагаемый:

1. **Явное указание количества** — "Количество заявок: 3" (высокий приоритет)
2. **Перечисление с номерами** — "Заявка №1, Заявка №2, Заявка №3"
3. **Нумерация в таблицах** — "1. ООО «Рога», 2. ООО «Копыта»"
4. **Уникальные ИНН** — подсчёт уникальных ИНН (низкий приоритет)

### 4.2. Новые паттерны для заявок

```python
# Паттерн: "заявка №N" с контекстом
APPLICATION_NUMBER_PATTERN = re.compile(
    r"(?:заявк[а-яё]*|заявление)\s*(?:№|#|N|n)\s*(\d+)",
    re.IGNORECASE
)

# Паттерн: таблица с номерами
TABLE_WITH_NUMBERS_PATTERN = re.compile(
    r"^\s*(\d+)\s*[.)]\s+",
    re.MULTILINE
)
```

## Этап 5: Обработка сценария "Несколько протоколов"

### 5.1. Архитектура для хранения состояния

```python
@dataclass
class MultiProtocolAnalysis:
    tender_id: str
    protocols: list[ProtocolData]
    all_applications: set[int]
    
    def add_protocol(self, protocol: ProtocolData):
        self.protocols.append(protocol)
        self.all_applications.update(protocol.application_numbers)
    
    def get_final_count(self) -> int:
        return len(self.all_applications)
```

## Критерии успеха

1. **Точность подсчёта:** 95%+ правильных определений количества участников
2. **Дедупликация:** Правильно объединяются заявки с одинаковыми номерами из разных таблиц
3. **Надёжность:** Логирование позволяет понять, как получен результат
4. **Расширяемость:** Легко добавить новые форматы протоколов

## Риски и их митигация

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Реальные протоколы не соответствуют ожиданиям | Высокая | Собрать тестовую выборку реальных протоколов до начала |
| Изменение формата на rostender | Средняя | Добавить абстракцию для лёгкой замены парсера |
| Производительность при большом объёме | Низкая | Асинхронная обработка, кэширование |

## Примеры для тестирования

Нужно собрать реальные протоколы для следующих сценариев:

1. **Одна таблица, один протокол** — простой случай
2. **Две таблицы в одном протоколе** — "Заявки" и "Допуск"
3. **Три протокола на один тендер** — разные даты
4. **Таблица без явных заголовков** — нужно искать по контексту
5. **Протокол без таблиц** — только текст
